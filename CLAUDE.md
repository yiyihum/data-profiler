## What This Project Is

Data Profiler is a state-machine-based automated data analysis pipeline. It takes a data directory (containing `description.md` + CSV/Parquet files), auto-discovers the data, infers the prediction task via LLM, runs four sequential analysis layers (L0-L3), and produces an MLE report, a preprocessing script, and visualization artifacts. LLM calls (OpenAI or Anthropic) drive the analysis at each layer; code generated by the LLM is executed inside an in-process sandbox.

## Commands

```bash
# Install dependencies
pip install -r requirements.txt

# Run the full pipeline (smoke test with demo data)
python main.py demo/ -o ./output -v

# Run with explicit LLM settings
python main.py <data_dir> --provider openai --model gpt-4o -o ./output

# Agent loop tuning flags
python main.py <data_dir> --max-turns-per-layer 5 --stop-threshold 0.80 --layer-timeout-sec 300

# Quick syntax check (no test suite exists yet)
python -m py_compile main.py core/*.py layers/*.py report/*.py
```

Environment variables: `OPENAI_API_KEY` or `ANTHROPIC_API_KEY` must be set.

### Input Directory Structure

The input is a directory containing:
- `description.md` — task description (competition page, business context, etc.)
- One or more CSV/Parquet data files (the largest or "train"-named file is auto-selected as primary)

No YAML config is needed. Target column, task type, and metric are inferred by the LLM from the description and data.

## Architecture

### Pipeline Flow

`main.py` orchestrates a linear L0 → L1 → L2 → L3 → Report pipeline. Each layer receives a shared `StateContext` and a `CodeSandbox`, produces structured JSON insights via LLM, and writes results back to state before advancing.

### Core Components (`core/`)

- **`state.py` — `StateContext`**: Dataclass holding all inter-layer state (`l0_stats`, `l1_insights`, `l2_selected_features`, `l3_strategy`, errors, artifacts, agent trace). Layers read prior state via `get_summary_for_layer()` to avoid context overflow. Serializes to JSON.
- **`sandbox.py` — `CodeSandbox`**: Executes LLM-generated Python in a shared `exec()` namespace with pandas/numpy/sklearn pre-imported. Persists `df` across calls. Has `execute_with_retry` (max 3 retries with optional LLM fix callback). Saves plots to `output_dir/artifacts/`.
- **`llm.py` — `LLMClient`**: Unified OpenAI/Anthropic client. OpenAI path uses `json_schema` response format for structured output; Anthropic path embeds schema in the system prompt and extracts JSON from the response. GPT-5 family uses `max_completion_tokens` instead of `max_tokens`.
- **`agent_loop.py` — `AgentLoop`**: Generic multi-turn loop used by every layer. Each turn calls `step_fn` → `evaluator_fn` (scores 0.0–1.0) → either converges (score >= threshold), retries, or falls back. Logs decisions and turn records into `StateContext`.
- **`evaluators.py`**: Per-layer scoring functions (`evaluate_l0_turn`, etc.) that weight different success signals.
- **`prompts.py`**: System prompts, JSON schemas (`L0_STATS_SCHEMA`, `L1_INSIGHTS_SCHEMA`, `L2_FEATURES_SCHEMA`, `L3_STRATEGY_SCHEMA`), and user-prompt builders for each layer.

### Layers (`layers/`)

Each layer class (e.g., `L0Cleaner`) follows the same pattern: defines `step_fn`, `apply_fn`, `fallback_fn`, and delegates to `AgentLoop.run()`.

- **L0 (`l0_cleaner.py`)**: Data quality — gathers `.info()`/`.describe()` stats, asks LLM for cleaning recommendations, executes high-confidence actions (drop columns, fill missing). Task config is intentionally hidden at this stage.
- **L1 (`l1_explorer.py`)**: Unsupervised exploration — distribution analysis, correlation matrix, skewness detection. Task config still hidden.
- **L2 (`l2_aligner.py`)**: Task-aligned analysis — receives `task_config` for the first time, computes feature-target relationships (mutual info), recommends transformations and feature selection.
- **L3 (`l3_strategist.py`)**: Model strategy — recommends models, preprocessing pipeline, and validation strategy.

### Report (`report/generator.py`)

`ReportGenerator` fills a Markdown template with data from final state. Also generates `layer_diagnostics.md` (convergence trace) and `preprocess_pipeline.py` (standalone script from L0 cleaning actions + L2 transformations).

## Key Conventions

- All state mutations go through `StateContext` methods — avoid ad-hoc globals.
- Layer classes follow the naming pattern `L{N}{Role}` (e.g., `L0Cleaner`, `L2Aligner`).
- LLM outputs are constrained by JSON schemas defined in `core/prompts.py`. When modifying a schema, update both the schema dict and the corresponding system/user prompt.
- Sandbox code must never modify original data files; it operates on in-memory DataFrame copies (`df`, `df_clean_v1`, etc.).
- Commit messages use conventional prefixes: `feat:`, `fix:`, `chore:`, `refactor:`.

## Output Artifacts

A successful run produces in the output directory:
- `pipeline_state.json` — full serialized state
- `agent_trace.json` — multi-turn agent decisions/scores
- `mle_report.md` — final analysis report
- `layer_diagnostics.md` — per-layer convergence summary
- `preprocess_pipeline.py` — executable preprocessing script
- `artifacts/` — generated plots (correlation matrix, feature importance, etc.)
